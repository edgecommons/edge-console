EdgeCommons · Low-Fidelity Wireframes · v0.2

Edge Console — a live, site-scoped operations UI for edgecommons components

One browser-facing console per site, aggregating every line → device → component over a Unified Namespace. It deploys as a standard edgecommons component and is the sole bridge between the browser (HTTPS + WebSocket) and the message bus (MQTT / IPC). These grayscale wireframes fix layout, hierarchy, and real-time behavior — not visual design. High-fidelity IBM Carbon mockups come next.

00 System context — one console, many device buses, one namespace

A site has multiple device buses (a local broker per HOST device; a device-local IPC bus per Greengrass core). A per-device uns-bridge relays each onto one site UNS broker; the console connects to that single bus and fans state out to every tab.

Why this shape (grounded). Greengrass IPC is device-local — two cores can't see each other's bus — and there's no MQTT retain and no consumer-side miss-detection in the platform. So aggregation happens below the console (the bridge), the console keeps its own last-known-value cache + staleness engine, and every tab gets a snapshot then deltas. On Kubernetes the in-cluster broker already is the site broker — no bridge needed inside a cluster (see DESIGN.md §3.6).

06 How a panel appears — and how rich control works — with no console rebuild

Everything travels over the bus (components aren't HTTP-reachable). Discovery is now standard verbs; control is confirmed request/reply.

1
State + Describe. Console sees the component's state keepalive, then requests cmd/describe → capability manifest incl. a panel manifest with multiple views (Overview / Address Space / Signals / Diagnostics) + the command catalog with JSON-Schema params.
2
Fetch over the bus. cmd/panel.get streams each descriptor (or signed WC bundle) in chunks; console verifies the hash (and ed25519 signature for WC) before use, caches content-addressed. 40 identical adapters transfer once.
3
Render multi-view. Sub-tabs per view. Descriptor views use console-owned Carbon widgets incl. treeBrowser + signalGrid; bespoke views use a signed, sandboxed Web Component with a descriptor fallback.
4
Rich control = mandated verbs. cmd/sb.browse (paged tree) · cmd/sb.read · cmd/sb.write (confirmed, with optional read-back). OPC-UA browse+read+write needs no Web Component.
5
Every write is safe + audited. RBAC → host-owned confirm modal (type-to-confirm) → audit before dispatch → adapter writes.allow[] → confirmed reply + read-back → Events mirror → site-broker ACLs scope cmd/# per device.
Fallback ladder — never a blank tab. WC fails/unsigned → its fallback descriptor → generic panel (heartbeat charts + observed metrics + topic tail). Every component has a useful page on day zero.
